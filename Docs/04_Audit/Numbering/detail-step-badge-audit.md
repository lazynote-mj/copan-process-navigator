# Process Detail Step Badge Audit

목적: 프로세스 상세 화면에서 원도형 순번을 모든 노드에 붙이지 않고, 사람이 실제로 처리하거나 판단하는 업무 단계에만 부여하기 위한 사전 검토안입니다.

작성 기준:
- 데이터 기준: `public/process-data/state.json`
- 프로세스그룹 기준: `src/data/toBeOverview/detail-process-groups.json`
- PDF 기준 보조자료: `Docs/scm-to-be-process-node-order.md`
- 이번 문서는 분석용입니다. 실제 JSON 데이터는 수정하지 않았습니다.

## 판단 기준

번호 부여 후보:
- 사람이 ERP/WMS/OMS/POS/그룹웨어/수작업으로 수행하는 업무
- 등록, 입력, 확인, 확정, 요청, 처리, 품의, 검토, 승인, 조회, 마감, 정산 같은 업무 행위
- 사람이 판단하는 의사결정 노드

번호 제외 후보:
- Database, API, Interface, System, Linked Process
- 재고현황, 출고정보, 입고정보, 입/출고정보, 주문정보처럼 상태나 저장소를 표시하는 노드
- 자동 연동, 자동 생성, 전표생성(미결), 전표생성(자동승인)처럼 시스템 결과에 가까운 노드
- 연결용 Process 노드

검토 필요:
- 위탁 여부, MG차감 여부, 재고차감처럼 사람이 판단하는지 시스템 Rule인지 불명확한 노드
- ERP/WMS/POS 노드 중 실제 현업 입력 단계인지 자동 처리 결과인지 불명확한 노드
- 예외사항처리, 집계, 마감등록처럼 업무 주체 확인이 필요한 정산 노드

## Legacy Detail 검토 대상

`business-to-project`는 현재 저장 데이터에는 존재하지만, 좌측 메뉴의 21개 프로세스그룹에는 직접 매핑되지 않는 legacy 상세 프로세스로 보입니다.

권장:
- 21개 프로세스그룹 순번 정리 후 유지/삭제/통합 여부를 별도 판단
- 이 프로세스에는 현재 `stepBadge`가 없으므로 이번 자동 반영 대상에서는 제외

## 반영 전 확인 질문

1. ERP/WMS/POS 화면에서 사람이 버튼을 눌러 수행하는 `등록/확정/입력/요청/마감` 단계는 모두 순번을 부여하는 방향으로 볼지?
2. `위탁 여부`, `MG차감 여부` 같은 판단 노드는 사람이 판단하는 업무인지, 시스템 Rule인지?
3. `전표생성(미결)`은 화면에 표시되더라도 자동 결과로 보고 순번을 제외할지?
4. `B2C 주문`, `이지어드민 주문수집`, `연동현황` 같은 외부/연동 이벤트는 순번 제외가 맞는지?

위 네 가지가 확정되면 다음 단계에서 `stepBadge`를 일괄 재부여하고, 번호가 없어야 하는 노드는 `0` 또는 미지정으로 정리할 수 있습니다.
